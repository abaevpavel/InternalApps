# 00-Платформа портала — Google-вход, User Management, Roles, доступы

> Модель: **vibe-coding (билдит Claude), в ЧАСАХ.** Это «нулевой уровень» — общий каркас, в
> котором живут все приложения портала «My Applications». Делается один раз.

## Простыми словами: кто что делает
- **Сотрудник** — заходит через **Google** (кнопка «Sign in with Google»).
- **Портал** — пускает внутрь **только если его email заранее заведён** админом (whitelist);
  показывает сетку приложений — но только те карточки, к которым у его роли есть доступ.
- **Админ (Pavel)** — в **User Management** заводит людей, назначает им роль; в **Roles**
  создаёт роли и галочками отмечает, какие приложения роль открывает.
- **База портала (Supabase)** — хранит users, roles, какие приложения какой роли доступны.
> Главное: портал — это **«проходная + раздача ключей»**. Сам он бизнес-логики приложений
> не делает; решает только «кого пускать» и «кто какие карточки видит».

---

## Архитектура: где живут приложения
- **Портал** = отдельный Supabase-проект (Lovable) + хостинг фронта. Здесь: Google-auth,
  whitelist, users/roles, доступы к приложениям, сетка «My Applications».
- **Каждое приложение** — это маршрут/SPA, который портал открывает по карточке. Часть
  приложений (HR-Checklists, Sales и т.п.) могут жить в том же портал-Supabase; **Task Planner —
  в СВОЁМ отдельном Supabase** («crews scheduling»), со своим auth и своими ролями.
- Портал-доступ = «показать карточку и пустить»; **роли ВНУТРИ приложения** (например
  team_lead в Task Planner) — отдельный механизм в проекте этого приложения.

> Решение по входу: **whitelist по email** (как Pavel) — любой Google-аккаунт логинится,
> но внутрь пускаем только заведённые email. Приложений к доступу: **6** (как на скрине).

---

## Scope (что строим)
- **Google OAuth** (Supabase Auth Google provider) + whitelist-гейт по email.
- **БД портала:** `users/profiles`, `roles`, `applications`, `role_applications`, `user_roles`.
- **RLS:** управлять users/roles может только Admin; обычный юзер видит только своё.
- **Экран User Management:**
  - вкладка **Users** — список (имя/email/роль/дата), поиск, сортировка, фильтр, добавить,
    редактировать роль, удалить.
  - вкладка **Roles** — список ролей с их приложениями, **Create Role**, edit (мультиселект
    приложений), delete.
- **«My Applications»** — сетка карточек, отфильтрованная по доступу роли.
- **My Account** — профиль/смена пароля (для Google-входа — базовый профиль).

## План (сессиями, ≈8 ч)
| # | Блок | Состав | Часы |
|---|------|--------|------|
| 1 | **Google-auth + whitelist** | Supabase Google provider, гейт «email есть в users → пускаем, иначе отказ» | 1.0 |
| 2 | **БД + RLS** | 5 таблиц (`users/profiles`, `roles`, `applications`, `role_applications`, `user_roles`) + политики (Admin управляет, юзер видит своё) | 1.5 |
| 3 | **Users-вкладка** | список + поиск/сортировка/фильтр + add/edit-role/delete + диалоги | 1.75 |
| 4 | **Roles-вкладка** | список ролей + Create/Edit Role (мультиселект из 6 приложений) + delete | 1.5 |
| 5 | **My Applications grid** | сетка карточек, фильтр по доступу роли; «Signed in as», меню (My Account / User Management / Sign out) | 1.0 |
| 6 | **My Account** | базовый профиль | 0.5 |
| 7 | **Wire 6 приложений** | строки в `applications` + карточки + ссылки «Open Application» | 0.25 |
| 8 | **QA + хостинг + деплой** | прогон ролей, домен, env | 0.75 |
| | **Итого** | | **≈ 8.25 ≈ 8 ч** |

**Вилка:** пол ~6.5 · реалистично ~8 · с трением ~11 (трение: Google-provider конфиг/redirect,
выверка RLS под Admin, мультиселект role↔applications).

> Это в основном **админский CRUD** — вайб-кодится быстро. Главный риск — не код, а
> **конфиг Google OAuth** (consent screen, redirect URIs, домен) и аккуратная RLS, чтобы
> не-админ не мог трогать users/roles.

---

## Дыры / на что заложить запас
- **Google consent / redirect** — внешний конфиг (Google Cloud), может сожрать время на домене.
- **RLS на users/roles** — критично: ошибка = эскалация прав. Тест под Admin/обычного юзера.
- **Per-app vs per-role доступ** — сейчас доступ на роль (роль→приложения). Если понадобится
  доступ «на конкретного юзера» сверх роли — +1–1.5 ч.

---

## Допущения и границы
- Вход — **whitelist по email**; приложений к доступу — **6**.
- Портал и Task Planner — **разные Supabase-проекты**; портал лишь открывает карточку TP.
- Хостинг портала — отдельная задача (входит в QA-блок: домен + env).
- Оценка — на каркас «как на скринах» (Users + Roles + Create Role + My Applications + My Account).
