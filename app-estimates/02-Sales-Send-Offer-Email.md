# 02-Sales — Send an Offer Email — план разработки и estimate

> Модель оценки: **vibe-coding (билдит Claude), в ЧАСАХ.** Оценка на честный rebuild до
> прод-стандарта. Стек: React + Vite + TS + Supabase. Переносим **только Lovable-часть**;
> сценарий Make.com + Airtable-сегментация получателей — внешний движок, вне scope.

---

## Что это
HTML-композер маркетингового «offer email» для отдела продаж. Письмо **не шлёт напрямую** —
постит payload в Make.com webhook; Make сам забирает получателей из Airtable и рассылает
от указанного отправителя. CRUD шаблонов, rich-text (Quill), до 3 дат-вариантов,
draft в localStorage, чистка HTML, 7 confirm-диалогов.

**Сложность: средняя.** Само приложение простое; к прод-уровню тянут безопасность
(публичный webhook, открытая RLS) и пара UX-багов.

## Простыми словами: кто что делает
- **Менеджер по продажам** — пишет письмо (тема, текст в редакторе, до 3 дат, «от кого»),
  сохраняет как шаблон или грузит готовый, жмёт «Отправить».
- **Приложение** — хранит шаблоны писем в базе, причёсывает HTML и отправляет «пакет»
  (тема + текст + даты + отправитель) в Make.
- **База (Supabase)** — хранит шаблоны писем.
- **Make (отдельно)** — берёт список получателей из Airtable, подставляет данные и **реально
  рассылает** письма.
- **Airtable** — список клиентов-получателей.
> Главное: приложение само письма **НЕ шлёт** и список получателей не знает — оно только
> готовит контент и «нажимает кнопку» у Make, а уже Make делает рассылку.

---

## Архитектура (как сейчас)
```
[/sales-email-sender] → CRUD email_templates (Supabase REST, RLS открыта)
                      → draft в localStorage
   Send → regex-чистка HTML → POST напрямую в Make.com webhook (публичный, без auth)
        → [Make подставляет даты/sender, берёт получателей из Airtable, шлёт] — ВНЕ репо
```
Своих edge functions нет.

---

## Scope
- **Перенос как есть:** 1 экран `/sales-email-sender` (composer + панель шаблонов + Send),
  таблица `email_templates` с открытой RLS, прямой POST в Make webhook, Quill, regex-чистка.
- **Исправления (отдельно):** edge `send-offer-email`, сужённая RLS, whitelist `email_senders`,
  аудит `email_send_log`, TipTap, DOMPurify, debounce, починка Radix Select.
- **Auth/доступ:** общие `applications → role_applications → user_roles`.

## Дыры / исправления (отдельно, поверх переноса)
Что в оригинале сломано/уязвимо/долг. Каждая — со своим временем. Делаются опционально.

| # | Дыра | Суть / чем грозит | Фикс | Prio | Часы |
|---|------|-------------------|------|------|------|
| F1 | **Публичный Make webhook из браузера** | URL зашит в bundle, POST без auth → кто знает URL, шлёт письма **от имени компании** | перенести в edge `send-offer-email`: JWT+роль, подпись `MAKE_WEBHOOK_SECRET` (секрет в проекте есть) | **P0** | 1.0 |
| F2 | **RLS открыта на `email_templates`** | все 4 операции `authenticated`; любой **удалит/изменит чужой шаблон** без аудита | UPDATE/DELETE — только владельцу/роли Admin/Sales-Lead | **P1** | 0.5 |
| F3 | **Сломанные rename/delete в Radix `Select`** | анти-паттерн (кнопки внутри `SelectItem`) → «плохо тыкаются», реальные жалобы | вынести в нормальную панель шаблонов | **P1** | 0.5 |
| F4 | **Нет debounce на Send** | `isLoading` ставится поздно → дабл-клик = **двойная рассылка батча** | дизейбл/loading сразу при подтверждении | **P1** | 0.25 |
| F5 | **Free-text sender без whitelist** | отправитель руками, без валидации (реально один) → опечатка = «не от того» | dropdown `email_senders` + email-валидация | P2 | 0.75 |
| F6 | **Quill устарел** | не дружит с React 18 strict mode, заморожен | миграция на TipTap | P2 | 1.0 |
| F7 | **HTML cleanup на regex** | ломает вложенность (`<p>` в `<li>`), нет санитайза → кривой HTML + XSS-поверхность | DOMPurify + email-safe нормализатор | P2 | 0.75 |
| F8 | **Мелочи** | `Update` не меняет `name`; старые шаблоны без `send_from` блокируют Send | поправить попутно | P3 | 0.25 |
| F9 | **Нет аудита отправок** *(фича-гэп)* | не видно кто/когда/что разослал | таблица `email_send_log` + просмотр | опц. | 1.0 |
| | **Итого исправления** | | | | **≈ 3.75 ч** (+1 опц. F9) |

> Суть долга: **точка отправки открыта** (F1) + **данные открыты на запись** (F2) + **два видимых
> бага** (F3 «не тыкается», F4 двойная рассылка). F5–F8 — гигиена редактора/HTML.
> **Безопасный минимум F1+F2+F3+F4 ≈ 2 ч.**

## Контракты и факты (из разбора Lovable)
- **Make webhook:** `https://hook.us1.make.com/413vwl0ap4pu2yjsiwiau6ppxoumnyv9`, POST JSON.
  Payload: `{subject, htmlContent, date1..3 ("EEEE, yyyy-MM-dd"|null), sendFrom, senderName}`. Без auth.
- **Airtable:** код не ходит; base/table/сегментация получателей — UNKNOWN (всё в Make).
- **БД:** `email_templates {id, name, subject, content(HTML), send_from?, sender_name?, created_by, created_at, updated_at}`. RLS — все 4 операции `authenticated`, без сужения. Объём: 5 шаблонов (1.2–2.3 KB HTML).
- **Secrets:** своих нет; общие `VITE_SUPABASE_*`. `MAKE_WEBHOOK_SECRET` в проекте есть (используется другими app), но это приложение его НЕ применяет.
- **Current-state:** работают CRUD/rename/load/draft/Quill/чистка/Send/7 диалогов/нормализация ввода.
  Слабые места: открытый webhook, открытая RLS, баг Radix Select, нет debounce, free-text sender, regex-HTML.

## Допущения и границы
- **Вне scope:** сценарий Make.com (сегментация/рендер/рассылка) и Airtable-база получателей — внешний движок, не трогаем.
- Перенос — **1:1 как есть** (Quill, прямой webhook, открытая RLS). Улучшения вынесены в «Дыры».
- TipTap, whitelist, аудит — отдельные исправления (F6/F5/F9), не входят в перенос.
