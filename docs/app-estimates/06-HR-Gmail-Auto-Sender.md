# 06-HR-Gmail Auto Sender — план разработки и estimate

> Модель оценки: **vibe-coding (билдит Claude), в ЧАСАХ.** Оценка на честный rebuild до
> прод-стандарта. Стек: React + Vite + TS + Supabase edge functions.

**⚠ Ключевой факт:** Lovable-часть тривиальна (1 страница + 1 крошечная edge function,
~190 LoC). Вся реальная сложность — в **AWS-сервисе вне этого репо** (Google OAuth клиент,
хранение refresh-токенов, отправка писем). Поэтому оценка дана в **двух границах scope.**

---

## Что это
Внутренняя утилита: HR вводит email нового корпоративного ящика → жмёт «Setup Gmail Auth»
→ открывается Google OAuth consent → после согласия upstream-сервис (на AWS) получает
право слать письма от имени этого ящика. Шаг онбординга «ENABLE EMAIL AUTO SENDER».

**Сложность Lovable-части: тривиальная.** Триггер-UI без БД, без истории, без ролевых
проверок внутри (доступ бинарный: открыл страницу → можешь нажать).

## Простыми словами: кто что делает
- **HR** — вводит email нового рабочего ящика и жмёт «Setup Gmail Auth».
- **Приложение** — передаёт этот email своей маленькой функции (edge function).
- **Edge function** — спрашивает у внешнего AWS-сервиса ссылку на «разрешение от Google».
- **AWS-сервис (чужой, НЕ мы)** — выдаёт ссылку, а позже хранит «ключ доступа» к ящику
  и реально рассылает письма от его имени.
- **Google** — показывает экран согласия; человек жмёт «разрешить».
> Главное: это просто **«кнопка запуска авторизации»**. Само приложение писем не шлёт и ключи
> не хранит — всю настоящую работу (хранение доступа, рассылку) делает внешний AWS-сервис.

## Архитектура (как сейчас)
```
[/gmail-auto-sender page] → supabase.functions.invoke('gmail-auth', {email})
        → edge function gmail-auth (verify_jwt=false)
        → GET https://3mb71kyw2k.execute-api.us-east-1.amazonaws.com/dev/gmail/auth?email=…
          header Authorization: bmasters2020   ← статичный токен, ЗАХАРДКОЖЕН
        → AWS возвращает {authentication_url}  → window.open(consent)
        → [AWS хранит refresh token, отправка писем — ВНЕ репо, UNKNOWN]
```

## Дыры / исправления (отдельно, поверх переноса)
Долг Lovable-части. Каждая — со своим временем. По каждой: суть → чем грозит → фикс.

| # | Дыра | Суть / чем грозит | Фикс | Prio | Часы |
|---|------|-------------------|------|------|------|
| F1 | **Токен `bmasters2020` в коде edge function** | статический secret в `gmail-auth/index.ts` → в каждом клоне и в git-истории; ротация только через релиз | в Supabase secret (`Deno.env.get`) | **P0** | 0.25 |
| F2 | **`verify_jwt = false`** | функцию дёргает любой, кто знает URL, без логина → обход портала, спам/abuse AWS-квоты | `verify_jwt = true` (фронт уже шлёт JWT) | **P0** | 0.1 |
| F3 | **Silent failure** | если ответ не-JSON, тостит «успех», но consent-вкладку не открывает → тихий баг онбординга | нет URL → явная ошибка, не success | **P1** | 0.25 |
| F4 | **AWS URL в коде (вкл. `/dev`)** | смена стейджа/ротация = правка кода | вынести в env (`…_AWS_URL`) | P2 | 0.1 |
| F5 | **Нет валидации email** | кривой адрес уходит в AWS/Google, отваливается там | клиентский regex-чек | P2 | 0.1 |
| F6 | **Нет аудита ящиков** *(фича-гэп)* | не хранит, какие ящики авторизованы → дубли/забывчивость | таблица-журнал `gmail_auth_log{email,by,at}`+RLS | опц. | 0.5 |
| | **Итого исправления** | | | | **≈ 0.8 ч** (+0.5 опц.) |

> Суть долга: **секрет и точка входа открыты** (F1/F2) + **сбой маскируется под успех** (F3).
> Обязательны F1+F2+F3; F4/F5 — копеечные попутно; F6 — опционально.

## Контракты и факты (из разбора Lovable)
- **AWS API Gateway:** `3mb71kyw2k.execute-api.us-east-1.amazonaws.com`, stage `dev`, path `/gmail/auth`.
  Auth — статичный header `Authorization: bmasters2020` (shared secret, не Bearer/IAM). Outbound only.
- **Google OAuth:** client_id/scopes/redirect живут в AWS-сервисе, **не в репо**. Scopes UNKNOWN (вероятно `gmail.send`). App в режиме unverified/testing.
- **БД:** нет. Единственная точка Supabase — вызов edge function. Доступ — общие `applications/role_applications/user_roles`.
- **Secrets:** собственных нет; AWS-токен захардкожен в `gmail-auth/index.ts`.
- **Current-state:** работает (страница → edge → URL → consent в новой вкладке). Партиал: AWS на `/dev` стейдже, Google app unverified. Половина «отправки» — вне репо, статус UNKNOWN.

## Допущения и границы
- **Вне scope (граница A):** весь AWS-сервис (OAuth-клиент, token storage, sending worker) — существующая внешняя система.
- Граница B предполагает полный переезд этой способности на Supabase + собственный Google Cloud проект.
- Google consent-screen verification — внешний процесс Google, не входит в часы разработки.
