# Портал «My Applications» — планы и estimate по приложениям

> Один документ на приложение. Модель оценки: **vibe-coding (билдит Claude), в ЧАСАХ.**
> Оценка на честный rebuild до прод-стандарта. Детали — в файле приложения.

## Методика
По каждому приложению оцениваем **отдельно**:
1. **Перенос «как есть»** — воспроизвести текущее прод-поведение Lovable-части 1:1 (чистым
   кодом, тот же функционал), **без** доделок и улучшений.
2. **Дыры / исправления** — отдельная таблица: каждая дыра (баг/уязвимость/долг) с приоритетом
   и **своим временем**. Делаются опционально, поверх переноса.

Итог приложения = `Перенос + (выбранные исправления)`. Внешние движки (Make/Airtable/AWS/n8n)
— вне часов, помечаются отдельно.

## Как добавляем новое приложение
1. Прогнать в Lovable: **deep-dive промпт** + **догоняющий по 4 пробелам**
   (интеграции / secrets / current-state / объёмы).
2. Создать `app-estimates/<app>.md` по шаблону:
   что это → архитектура → **Перенос «как есть» (план+часы)** → **Дыры/исправления (таблица+часы)**
   → контракты/факты → границы.
3. Обновить строку в сводной таблице (Перенос / Исправления / Итого).

## Принципы оценки (чтобы цифрам можно было доверять)
- **Анкор — это разбивка по блокам, а не итог.** Итог = сумма блоков; двигаем конкретные
  строки, а не общий множитель.
- **Vibe-coding сжимает генерацию кода, НЕ сжимает:** итерации RLS, тюнинг AI-частей,
  edge-кейсы dnd/undo, ручное QA, деплой, координацию с внешними системами (Make/Airtable/n8n).
- Внутренности внешних сценариев (Make/Airtable/n8n) — **вне часов приложения**, оцениваются отдельно.
